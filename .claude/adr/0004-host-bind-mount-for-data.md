# ADR-0004: 데이터는 호스트 바인드 마운트에 둔다

- 상태: Accepted
- 관련: [ADR-0005](0005-engine-based-backup.md), `scripts/start.ps1`, `scripts/down.ps1`

## 배경 (Context)

DB 데이터는 컨테이너 수명과 무관하게 살아남아야 한다. 컨테이너를 재생성하거나
이미지를 올리는 일은 잦지만, 그때마다 데이터가 날아가면 안 된다. 또한 기존 SQL Server의
데이터 폴더를 그대로 옮겨 붙이는 이관 시나리오도 지원해야 한다.

## 결정 (Decision)

데이터를 **호스트 바인드 마운트**로 저장한다: `DATA_ROOT/<PREFIX_DIR>/data` →
컨테이너 내부 `/var/opt/mssql/data`. named volume 이 아니라 호스트의 실제 경로를 쓴다.

- `start.ps1`은 **기동 전에 데이터 폴더를 먼저 만든다.** 폴더가 없는 채로 올리면 Docker가
  빈 폴더를 생성해 기존 DB 를 못 붙기 때문이다.
- `down.ps1`은 컨테이너/네트워크만 제거하고 **데이터는 건드리지 않는다.**
- 컨테이너명(`_NAME`)과 폴더명(`_DIR`)을 다르게 둘 수 있어(예: `_DIR=Db2019A-old`) 기존
  데이터 경로 호환 이관이 가능하다.

## 결과 (Consequences)

- **좋은 점**: `down` → 이미지 갱신 → `start` 사이클에도 데이터가 보존된다.
- **좋은 점**: 데이터가 호스트 파일시스템에 있어 백업·이관·용량 확인이 직관적이다.
- **감수할 점**: 폴더 선생성 규약을 지켜야 하며(스크립트가 자동화), `DATA_ROOT` 상위 경로/드라이브는
  미리 존재해야 한다.
- **감수할 점**: 현재 마운트는 `data`만이다. errorlog(`/var/opt/mssql/log`)와 secrets 는
  컨테이너와 함께 사라진다(로그는 `logs.ps1 -ErrorLog`로 실행 중에만 확인 가능).
